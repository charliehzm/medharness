# MedHarness Console · 产品设计文档

> **状态**：v0.6 产品设计（draft）· 配套 [unified-gateway.md](../architecture/unified-gateway.md) §5
> **定位**：把"网络层合规网关"变成客户**看得见、改得动、信得过**的产品界面
> **配套原型**：[prototype/console-demo.html](../../prototype/console-demo.html)（单文件可点击 · 浏览器直开 · 销售/融资 demo）

---

## 0. 一句话产品定位

> MedHarness Gateway 在网络层拦下每一条含 PHI 的 LLM 流量；MedHarness Console 让 CTO 一眼看懂合规健康、让合规官 4 小时交出监管包、让工程师改阈值不碰 YAML。**网关是护城河，Console 是让护城河被看见的窗户。**

---

## 1. 目标用户与核心诉求（Job-to-be-Done）

| 角色 | 一句话画像 | 最怕什么 | 来 Console 要什么 | 默认首屏 |
|---|---|---|---|---|
| **CTO / 技术负责人** | 为合规结果背锅的人 | 监管罚款 4% 营业额 · 不知道自己漏没漏 | **一个能信的数字**：今天合规吗？ | 🏠 合规态势 |
| **合规官 / Compliance Officer** | 监管来了第一个被问的人 | 4 小时交不出审计包 | 任意一条 PHI 的**完整血缘 + 哈希链** | 🔍 审计追溯 |
| **工程师 / Harness Eng** | 配阈值、接上游、修红灯的人 | 改错配置开了后门 | **不碰源码**改策略 + 看流量 | 🚦 研发流水线 |

**设计含义**：三类用户三套默认视图，但共享同一份数据真相。登录即落到"我的"页，而不是逼用户在导航里找。

## 2. 六条设计铁律（5 易用性 + 1 安全）

| # | 铁律 | 为什么 | 在原型里怎么落地 |
|---|---|---|---|
| 1 | **一个数字定生死** | CTO 没空看 20 个指标 | 首屏最大元素 = 合规健康分 0-100 + 红黄绿环 |
| 2 | **角色化默认视图** | 不同人来要的东西不同 | 顶部角色切换 → 默认页随之变 |
| 3 | **渐进披露三层** | 一屏塞满 = 没人看 | 概览卡片 → 点击钻取 → 详情抽屉 |
| 4 | **零术语门槛** | "PHI recall 1.0" 业务听不懂 | 全部翻译成"患者信息泄漏拦截率 100%" |
| 5 | **每个红灯给出路** | 报警不给方案 = 焦虑制造机 | 每个红灯卡片带 `[点这里看怎么修]` → runbook |
| 6 | **🛡️ 永不显示原始 PHI** | Console 自己不能成泄漏点 | 顶栏常驻"本页 0 PHI"绿标 · 全程只显示占位符 + 哈希 |

> **铁律 6 是红线**：Console 是合规产品，它的界面绝不能反过来成为 PHI 泄漏面。一切展示的是**脱敏后占位符 + 元数据 + 哈希指纹**，从不展示 `张三 110101199001011234`，只展示 `__NAME_a1__ __ID_b2__` 和 `routing#a1b2`。

## 3. 信息架构

```
MedHarness Console
├── 🏠 合规态势      ← CTO 落点 · 健康分 + 7 日趋势 + 红灯清单
├── 📊 流量监控      ← 杀手锏 · PHI 桑基图 + 实时事件流（开发期/生产期分层）
├── 🔍 审计追溯      ← 合规官落点 · 血缘图 + 哈希链 + 一键监管包
├── 🚦 研发流水线     ← 工程师落点 · change 看板（仅开发期）
├── ⚙️  策略配置      ← 配置中心 · 6 类配置 + diff 预览 + 审批流
├── 🔌 上游接入      ← 接入向导 · 3 步生成 base_url + key
└── 📋 报表中心      ← 季度合规报表 + 监管应对包导出
```

**导航即权限**：工程师看不到"提交审批"按钮（只能发起），合规官看不到"接入新上游"（只读流量）。导航项按角色高亮，不可用项灰显而非隐藏（让用户知道"有这个能力，只是我没权限"）。

## 4. 核心页面设计

### 4.1 🏠 合规态势（CTO 落点）

**唯一目的**：让 CTO 在 3 秒内回答"我今天合规吗"。

```
┌──────────────────────────────────────────────────────────────┐
│  合规健康分                                                     │
│        ╭─────────╮                                             │
│        │   98    │  🟢 健康          较昨日 +2                  │
│        │ / 100   │                  本周 0 次 PHI 漏出           │
│        ╰─────────╯                                             │
│                                                                │
│  ┌─ 闸门一 ─┐ ┌─ 闸门二 ─┐ ┌─ 闸门三 ─┐ ┌─ 第四层 ─┐          │
│  │ PHI扫描  │ │ 脱敏     │ │ 路由     │ │ 注入防御 │          │
│  │ 🟢 100%  │ │ 🟢 正常  │ │ 🟢 11/11 │ │ 🟢 100%  │          │
│  │ 拦截率   │ │ p99 0.02ms│ │ 越权全防 │ │ 阻断率   │          │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘          │
│                                                                │
│  ⚠️ 需要注意（1）                                              │
│  🟡 dev-codex 上游今日触发限流 30 次 → [点这里看怎么修]        │
└──────────────────────────────────────────────────────────────┘
```

**设计要点**：
- 健康分是页面**绝对最大**元素，红黄绿一眼定性
- 四道闸门=四张状态卡，每张一个"翻译过的"数字（拦截率，不是 recall）
- 红灯/黄灯区永远在健康分正下方，且**每条带出路链接**
- 健康分算法对外可解释：`100 − 漏出×50 − 高危配置未审批×10 − 闸门降级×5`（让 CTO 信任这个数字怎么来的）

### 4.2 📊 流量监控（产品杀手锏）

**这是销售/融资 demo 的核心**——让客户**亲眼看到**自己的 PHI 被一层层拦下来，比任何 PPT 都有力。

**两个组件**：
1. **PHI 桑基图**（最近 1 小时聚合）：上游 → 闸门 → 下游，流量带宽=请求量，颜色=处置（绿=放行/脱敏后，红=阻断）
2. **实时事件流**：每条一行，🟢 放行 / 🟡 脱敏 / 🔴 阻断，带 `routing#xxxx` 可点进审计

```
上游            闸门                          下游
┌─────────┐
│Dify RAG │━━━━┓                            ┏━━→ 私有 Qwen (脱敏)
│ 1.2k/h  │    ┃  ┌──────────────────┐      ┃
└─────────┘    ┣━→│ PHI 扫描         │━━━━━━┫
┌─────────┐    ┃  │ 命中 47 → 脱敏   │      ┃
│ComfyUI  │━━━━┫  │ 拦截 3 → 阻断 🔴 │      ┗━━→ 境外 (已脱敏 12)
│ 380/h   │    ┃  └──────────────────┘
└─────────┘    ┃                              ✗ 阻断 3（未脱敏 L4）
┌─────────┐    ┃
│Codex dev│━━━━┛
│ 90/h    │
└─────────┘

实时事件流（每 2-3 秒进一条）
🟢 14:23:01 dify-rag    L2 脱敏后路由 qwen-max    routing#a1b2
🔴 14:23:00 comfyui     L4 未脱敏 → 阻断          阻断#c3d4
🟢 14:22:58 codex-dev   无 PHI 直接路由           routing#e5f6
```

**设计要点**：
- 顶部 `[全部] 开发期 生产期` 三态过滤——让客户看到"开发期和生产期同一个网关"
- 桑基图的"流"用**动画粒子**沿路径流动（live 感），阻断的粒子撞到红色闸门停下
- 事件流**自动滚动 + 新条目高亮淡入**，红色阻断事件抖动一下吸引注意
- 全程**只显示脱敏后占位符**——demo 时客户问"那张三呢？"答："桑基图上你永远看不到张三，那是设计——闸门二在第④步就把他加密了，Console 只看得到 `__NAME_a1__`"

### 4.3 🔍 审计追溯（合规官落点）

**唯一目的**：监管说"回放 2026-03 那条 PHI 调用"，4 小时内交包。

```
┌──────────────────────────────────────────────────────────────┐
│ 审计追溯           搜索 [routing#a1b2______] 🔍   [导出监管包]  │
├──────────────────────────────────────────────────────────────┤
│  血缘图（一条 PHI 走完全程）                                    │
│                                                                │
│  上游请求 ──→ 闸门一命中 ──→ 闸门二加密 ──→ 闸门三路由 ──→ 下游 │
│  dify-rag     PHI×2        AES-256-GCM    qwen-max     私有     │
│  14:23:01     姓名/身份证   __NAME_a1__   异构✓        ✓        │
│     │            │             │            │           │      │
│     └────────────┴─────────────┴────────────┴───────────┘      │
│              哈希链 ✓ 完整（block #18,432）                     │
│                                                                │
│  调用详情                                                       │
│  • prompt（脱敏后）: "帮我看下 __NAME_a1__ __ID_b2__ 的..."     │
│  • 模型: qwen-max-2026-01 (私有部署)                           │
│  • 数据分级: L3 · allowlist ✓ · 异构性 ✓                       │
│  • 当前哈希: 9f3a...c4d2  ← 上一条: 7b1e...a8f0                 │
└──────────────────────────────────────────────────────────────┘
```

**设计要点**：
- 血缘图横向 5 段对应 4 闸门数据流，每段可点开详情抽屉
- `[导出监管包]` 一键生成 AUDIT_BUNDLE（prompt history / model versions / routing decisions / 哈希链）
- 哈希链断裂 → 整页顶部红条 "⚠️ 哈希链在 block #X 断裂 · 已自动告警"
- **仍然 0 PHI**：prompt 显示的是占位符版，原始反查需走受控环境单独授权（界面外）

### 4.4 ⚙️ 策略配置（配置中心）

**唯一目的**：合规官/工程师**不碰 YAML**改策略，且改不出后门。

```
┌──────────────────────────────────────────────────────────────┐
│ 策略配置                              [YAML 视图] [保存草稿]    │
├──────────────────────────────────────────────────────────────┤
│ ▸ 上游接入   │  生产环境策略 (context=prod)                    │
│ ▾ 场景策略   │  ┌──────────────────────────────────────┐      │
│   • dev      │  │ PHI 命中处理                          │      │
│  ▶ prod ←    │  │  ○ 仅警告   ● 硬阻断   ○ 警告+脱敏    │      │
│ ▸ 模型白名单 │  │                                      │      │
│ ▸ 闸门阈值   │  │ 数据分级强制检查                      │      │
│ ▸ PHI 字段   │  │  ☑ L3   ☑ L4   ☐ L2                  │      │
│ ▸ 保留策略   │  │                                      │      │
│              │  │ 注入防御     ● 阻断   ○ 警告          │      │
│              │  └──────────────────────────────────────┘      │
│              │                                                 │
│              │  ⚠️ 此变更影响 2 个生产上游 (dify/comfyui)      │
│              │     高危变更 → 需合规委员会会签 → [提交审批]    │
│              │                                                 │
│              │  变更预览 (diff):                               │
│              │   - block_on_injection: warn                   │
│              │   + block_on_injection: block                  │
└──────────────────────────────────────────────────────────────┘
```

**配置 UX 三原则**（防止"配置本身成为合规风险"）：
1. **改前看影响**："此变更影响 2 个生产上游" → 防误操作
2. **diff 预览**：改了什么一目了然 · 像 code review
3. **嵌入审批流**：高危配置（policies/allowlist/retention）改完**不直接生效** → 走会签 → 落审计 → 可回滚

**配置治理矩阵**（界面据此决定显示"保存"还是"提交审批"）：

| 配置类 | 改动权限 | 审批 | 按钮文案 |
|---|---|---|---|
| upstreams / rate_limit | 技术委员会 | 单签 | `[保存]` |
| policies / gates 阈值 | 技术委员会 | + 合规会签 | `[提交审批]` |
| allowlist 加模型 | 技术委员会 | + 合规会签 | `[提交审批]` |
| fields PHI 字段 | 合规委员会 | 单签 | `[保存]` |
| retention 缩短保留期 | 双委员会 + 法务 | 三签 | `[提交三签审批]` |

### 4.5 🔌 上游接入

**唯一目的**：客户照贴 base_url + key 即可接入，零代码改造。

```
┌──────────────────────────────────────────────────────────────┐
│ 上游接入                                        [+ 接入新上游]  │
├──────────────────────────────────────────────────────────────┤
│ 名称            场景  协议      状态   今日流量  PHI 拦截        │
│ prod-dify-rag   prod  openai    🟢健康  8.2k     命中312/拦5    │
│ prod-comfyui    prod  openai    🟢健康  2.1k     命中88/拦2     │
│ dev-claude-code dev   anthropic 🟢健康  340      命中12/拦0     │
│ dev-codex       dev   openai    🟡限流  890      命中30/拦0     │
└──────────────────────────────────────────────────────────────┘
```

**接入向导 3 步**：选协议（openai/anthropic）→ 选场景（dev/prod）→ 生成 `base_url + API key` → 复制给 Dify/ComfyUI/IDE。

## 6. 关键交互流程

### 6.1 demo 故事线（销售/融资 90 秒脚本）
1. 落 **合规态势** → "98 分，绿的，今天没漏" （铁律 1）
2. 切 **流量监控** → 桑基图粒子流动 → "这是你公司此刻真实的 LLM 流量" （杀手锏）
3. 指红色阻断 → "ComfyUI 刚想把一张没脱敏的 L4 影像发出去，被闸门挡了" （看得见的价值）
4. 客户问"那患者姓名呢" → 指占位符 → "你在我界面上永远看不到张三，这是设计" （铁律 6 = 信任）
5. 切 **审计追溯** → `[导出监管包]` → "监管来了，4 小时这一包交出去" （合规官买单点）
6. 切 **策略配置** → 改个开关 → diff + "需会签" → "改配置像 code review，改不出后门" （CTO 安心点）

### 6.2 红灯 → runbook
任意红/黄卡片 → `[点这里看怎么修]` → 跳对应 runbook（如限流 → `08-troubleshooting`）。

### 6.3 配置审批闭环
改高危配置 → diff 预览 → `[提交审批]` → 进审批队列 → 合规官会签 → 生效 + 落审计 → 出问题可回滚到任意历史版本。

## 7. 数据边界与安全约束（红线）

| 约束 | 落地 |
|---|---|
| **永不显示原始 PHI** | 全界面只显示占位符 `__NAME_a1__` + 哈希 `routing#xxxx` + 聚合数（命中 47） |
| **本页 0 PHI 绿标** | 顶栏常驻 · 给客户/审计可见的承诺 |
| **原始反查需授权** | 反查走受控环境单独授权流，**不在 Console 界面内** |
| **Console 操作落审计** | 看了哪条审计、改了什么配置、导出哪个包 → 全部落 audit-log |
| **角色最小权限** | 工程师不能会签、合规官不能接上游 · 导航灰显不可用项 |

## 8. HTML 原型说明

**文件**：[prototype/console-demo.html](../../prototype/console-demo.html)（单文件 · 内联 CSS/JS · 零依赖 · 浏览器双击直开）

**已实现视图**：
- ✅ 🏠 合规态势（健康分圆环 + 4 闸门卡 + 红灯清单）
- ✅ 📊 流量监控（**SVG PHI 桑基图 + 动画粒子流 + 实时事件流自动滚动**）
- ✅ 🔍 审计追溯（血缘图 + 哈希链 + 模拟导出监管包）
- ✅ ⚙️ 策略配置（开关控件 + 实时 diff 预览 + 影响提示 + 审批按钮态切换）
- ✅ 🔌 上游接入（上游表 + 状态）
- ✅ 角色切换（CTO / 合规官 / 工程师 → 默认视图随之变）
- ✅ 顶栏"本页 0 PHI"常驻绿标

**用途**：销售现场 / 投资人 demo / 客户 POC 预演。无需部署、无需后端，所有数据为合成 mock（绝无真实 PHI）。

**技术说明**：纯静态。桑基图用 SVG `path` 手绘 + CSS `stroke-dashoffset` 动画做粒子流；实时事件流用 `setInterval` 注入合成事件；健康分圆环用 SVG `stroke-dasharray`。无外部 CDN，离线可用。
